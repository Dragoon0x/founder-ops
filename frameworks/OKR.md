# OKR Framework

## The Format

**Objective:** Qualitative, ambitious, inspiring (the "what")
**Key Results:** Quantitative, measurable, time-bound (the "how we know")

```
OBJECTIVE: [Aspirational statement]
  KR1: [Measurable outcome] — current X → target Y by [date]
  KR2: [Measurable outcome] — current X → target Y by [date]
  KR3: [Measurable outcome] — current X → target Y by [date]
```

## Worked Example (Quarterly OKRs)

```
COMPANY OKR Q2 2026

O1: Land our first enterprise wedge
  KR1: Close 3 enterprise deals (>$50K ARR each) — current 0 → target 3
  KR2: Hit $100K ARR from enterprise — current $0 → target $100K
  KR3: Reduce enterprise sales cycle from 90 days → 60 days

O2: Build the foundation for scalable retention
  KR1: Improve net retention from 105% → 115%
  KR2: Reduce monthly churn from 4% → 2%
  KR3: Increase 90-day activation from 55% → 75%

O3: Establish ourselves as a thought leader
  KR1: Publish 12 long-form pieces (currently averaging 4/quarter)
  KR2: Grow newsletter from 2K → 8K subscribers
  KR3: Land 3 podcast appearances on shows with 10K+ listeners
```

## Rules

### Quantity
- 3-5 Objectives max per quarter
- 3-5 Key Results per Objective
- Less is more (3 + 3 + 3 is plenty)

### Timeframe
- Quarterly (most common)
- Annual for company-level
- Don't change mid-quarter (kills the system)

### Difficulty
- Aim for "ambitious" — 70% completion = success
- 100% completion = you set them too easy
- 30% completion = you set them too hard
- Don't tie to compensation (compromises aspirational nature)

### Cascade
- Company OKRs → Team OKRs → Individual OKRs
- Each level should clearly contribute to the level above
- Not every team needs OKRs (small teams can use company OKRs)

## What Makes a Good OKR

### Good Objective
- ✅ "Become the default tool for engineering teams in fintech"
- ❌ "Grow revenue" (too vague)
- ❌ "Increase MRR by 30%" (this is a KR, not an O)

### Good Key Result
- ✅ "Increase activation rate from 55% to 75% by June 30"
- ❌ "Improve activation" (not measurable)
- ❌ "Ship the new onboarding flow" (this is a task, not a result)

### Difference: Tasks vs Results
- TASK: What you'll do
- RESULT: What will be different because you did it

| Task (avoid as KR) | Result (good KR) |
|-------------------|------------------|
| Hire 3 engineers | Engineering velocity up 50% |
| Launch enterprise tier | $100K ARR from enterprise |
| Redesign onboarding | Activation rate hits 75% |
| Build SOC 2 | Pass first enterprise security review |

## OKR Cadence

| Cadence | Activity |
|---------|----------|
| Weekly | Check-in: are we on track? Any blockers? |
| Monthly | Confidence review: 1-10 on hitting each KR |
| End of quarter | Score (0-1.0) each KR, retro |
| Start of quarter | Set new OKRs based on retro + strategy |

## Scoring at End of Quarter

| Score | Meaning |
|-------|---------|
| 1.0 | Hit the target exactly |
| 0.7-0.9 | Strong success |
| 0.5-0.7 | Meaningful progress |
| 0.3-0.5 | Some progress, fell short |
| 0.0-0.3 | Failed |

Average score across all KRs = quarterly performance.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Tasks as KRs | Rewrite as outcomes |
| 10+ OKRs | Cut to 3-5 |
| Sandbagging (easy KRs) | Push for "ambitious" — 70% target |
| Too hard (everyone fails) | Recalibrate next quarter |
| Setting then ignoring | Weekly check-ins |
| Tying to comp | Compromises ambition |
| OKRs that don't connect to strategy | Re-derive from strategy |
| Changing OKRs mid-quarter | Don't — set new ones next quarter |
